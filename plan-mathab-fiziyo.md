# پلن جامع پروژه — وب‌سایت و پنل مدیریت مطب دکتر کیمیا حبیبی سیاهپوش

## ۱. خلاصه مدیریتی (Business Plan)

**نام پروژه:** پلتفرم آنلاین مطب فیزیوتراپی دکتر کیمیا حبیبی
**نوع فعالیت:** وب‌سایت معرفی/خدمات + پنل مدیریتی چندنقشه (Admin / Doctor / Secretary)
**هدف اصلی:** جذب بیمار از طریق سئوی محلی، نمایش حرفه‌ای خدمات، و مدیریت داخلی نوبت‌دهی و فرم‌ها بدون نیاز به نرم‌افزار جداگانه.
**هدف ثانویه:** پایه‌ای آماده برای تبدیل بعدی به اپ موبایل (iOS/Android) با React Native، با همان API.

**مزیت رقابتی:**
- سئوی تخصصی روی نام دکتر و کلمات کلیدی درمانی (فیزیوتراپی رباط‌کریم، طب فیزیکی و توانبخشی، PRP، شاک ویو و ...)
- پنل اختصاصی برای منشی که کار نوبت‌دهی را به‌صورت دیجیتال و بدون خطا انجام می‌دهد
- هزینه زیرساخت تقریباً صفر (Cloudflare Free/Workers Paid حداقلی) چون فقط داده متنی ذخیره می‌شود

**مدل درآمدی غیرمستقیم:** افزایش تعداد نوبت از طریق سایت + کاهش هزینه نیروی انسانی پاسخ‌گویی تلفنی.

## ۲. ساختار صفحات سایت (Public Website)

| صفحه | محتوا |
|---|---|
| خانه | معرفی کوتاه دکتر، خدمات برجسته، CTA رزرو نوبت، نظرات بیماران |
| درباره ما | بیوگرافی کامل، تحصیلات، رتبه بورد، سابقه ۵۰۰۰+ بیمار |
| خدمات | لیست کامل خدمات (EMG، طب سوزنی، فیزیوتراپی، لیزر پرتوان، تکارتراپی، شاک ویو، مانیپولاسیون، ورزش درمانی، PRP، ژل، ازون، بوتاکس) هرکدام با صفحه اختصاصی برای سئو |
| تماس با ما | آدرس (رباط‌کریم)، نقشه، تلفن، ساعات کاری |
| رزرو نوبت | فرم گرفتن وقت آنلاین (نام، تلفن، نوع خدمت، تاریخ/ساعت پیشنهادی) |
| بلاگ (اختیاری، فاز ۲) | مقالات سئو-محور درباره دردهای اسکلتی-عضلانی |

**دوزبانه:** فارسی (RTL، پیش‌فرض) + انگلیسی (LTR) با سوئیچر زبان در هدر.
**تم:** Light theme با پالت آرامش‌بخش پزشکی (آبی روشن/سبز نعنایی + سفید)، گرادینت‌های ملایم mesh، تایپوگرافی فارسی خوانا (مثلاً Vazirmatn).

## ۳. پنل مدیریتی (Admin Panel) — ساب‌دامین `portal.example.com`

### نقش‌ها و سطح دسترسی (RBAC)

| نقش | دسترسی |
|---|---|
| **Developer (سوپرادمین)** | دسترسی کامل: زیرساخت، تنظیمات فنی، مدیریت کاربران، دیتابیس، لاگ‌ها، دیپلوی |
| **Doctor** | دسترسی به همه بخش‌های محتوایی و عملیاتی سایت (نوبت‌ها، بیماران، فرم‌ها، گزارش‌ها، تنظیمات محتوای سایت) — **بدون** دسترسی به بخش فنی/زیرساخت/کد |
| **Secretary (منشی)** | دسترسی محدود: ثبت/ویرایش داده بیماران، پر کردن فرم‌ها، تنظیم/مدیریت نوبت‌ها و بوکینگ‌ها — بدون دسترسی به تنظیمات سایت یا گزارش‌های مالی/مدیریتی |

### امنیت ورود
- ورود با ایمیل + رمز عبور (هش با Argon2/bcrypt)
- **احراز هویت دومرحله‌ای (2FA) از طریق ایمیل**: کد یک‌بارمصرف به ایمیل خود کاربر (Developer/Doctor/Secretary) ارسال می‌شود
- Session/JWT با Httponly + Secure cookie، rate-limiting روی فرم لاگین (جلوگیری از Brute-force)
- Audit log برای تمام عملیات حساس (چه کسی، چه زمانی، چه تغییری)

## ۴. زیرساخت فنی (Tech Stack نهایی)

| لایه | فناوری |
|---|---|
| فرانت‌اند + بک‌اند | **Next.js (App Router) + TypeScript + React** |
| استایل | Tailwind CSS |
| دیتابیس رابطه‌ای | **Cloudflare D1** (SQLite در edge) — برای کاربران، نوبت‌ها، فرم‌ها، بیماران |
| کش/سشن سریع | **Cloudflare KV** — برای session، rate-limit counter، کش محتوای عمومی |
| دیپلوی | **Cloudflare Workers** از طریق آداپتور **OpenNext** (`@opennextjs/cloudflare`) — جایگزین مدرن و توصیه‌شده‌ی Cloudflare به‌جای `next-on-pages` قدیمی |
| ایمیل (2FA و اعلان نوبت) | Cloudflare Email Workers یا سرویس ایمیل ترنزکشنال (مثل Resend/MailChannels) |
| نسخه‌ی موبایل (فاز ۲) | React Native، با مصرف همان API های Next.js (بدون نیاز به بازنویسی بک‌اند) |

### پاسخ به سؤال فنی شما:
بله — **Next.js کاملاً روی Cloudflare و با D1 + KV قابل استفاده است.** روش رسمی و فعلی Cloudflare استفاده از آداپتور OpenNext است که Next.js را به یک Cloudflare Worker واقعی تبدیل می‌کند (نه فقط استاتیک)، از SSR/ISR/Middleware پشتیبانی می‌کند و بایندینگ‌های D1/KV/R2 را مستقیم در دسترس کد سرور قرار می‌دهد. این دقیقاً همان چیزی است که برای این پروژه (لاگین، نقش‌ها، فرم‌ها، 2FA) نیاز است.

## ۸. ماژول اخبار / RSS

| بخش | شرح |
|---|---|
| منبع خبر | Aggregation از فیدهای RSS معتبر پزشکی/فیزیوتراپی (داخلی و بین‌المللی، مثل منابع خبری وزارت بهداشت، PubMed news، Physiotherapy journals) |
| نمایش در لندینگ | ویجت «آخرین اخبار» با ۳ تا ۴ خبر برتر، آپدیت خودکار |
| صفحه‌ی اخبار مجزا | `/news` با لیست کامل، فیلتر بر اساس موضوع (ایران/جهان)، صفحه‌بندی |
| پیاده‌سازی فنی | Worker زمان‌بندی‌شده (**Cloudflare Cron Trigger**) که فیدهای RSS را هر چند ساعت Fetch و Parse کرده و در جدول `news_items` ذخیره می‌کند؛ صفحات از D1 می‌خوانند (نه از فید زنده) تا سرعت و کنترل کیفیت حفظ شود |
| کنترل کیفیت | امکان حذف/تأیید خبر توسط Doctor یا Developer قبل از نمایش عمومی (پرچم `is_approved`) |

جدول اضافه به دیتابیس:
```
news_items (id, title, summary, source_name, source_url, category[iran|world], published_at, is_approved, fetched_at)
rss_sources (id, name, feed_url, category, active)
```

## ۹. کنترل نسخه، ریلیز و CI/CD

| بخش | شرح |
|---|---|
| مخزن کد | Git روی **GitHub** (Private Repository) |
| استراتژی برنچ | `main` (پروداکشن) ← `develop` (استیجینگ) ← `feature/*` |
| نسخه‌بندی | **Semantic Versioning** (`v1.0.0`, `v1.1.0`, ...) + Git Tag برای هر ریلیز |
| امضای دیجیتال ریلیز | فعال‌سازی **GPG/SSH Commit & Tag Signing** — هر تگ ریلیز با کلید GPG امضا می‌شود تا یکپارچگی و اصالت کد تضمین شود (`git tag -s vX.Y.Z`) و GitHub به‌صورت "Verified" نمایش می‌دهد |
| مدیریت Secrets | تمام کلیدها (Cloudflare API Token، کلیدهای ایمیل و ...) در **GitHub Encrypted Secrets**، هرگز در کد |
| CI/CD | **GitHub Actions** workflow شامل: ۱) Lint + TypeCheck ۲) Test ۳) Build با OpenNext ۴) Deploy خودکار به Cloudflare Workers هنگام push/merge به `main` ۵) اجرای D1 migrations |
| محیط‌ها | Staging (پیش‌نمایش روی ساب‌دامین `staging.portal...`) و Production جدا از هم |

نمونه ساختار Workflow (`.github/workflows/deploy.yml`):
```yaml
name: CI/CD
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20 }
      - run: npm ci
      - run: npm run lint && npm run typecheck
      - run: npm run test
      - run: npx wrangler d1 migrations apply DB --remote
        if: github.ref == 'refs/heads/main'
      - run: npm run deploy   # opennextjs-cloudflare build && wrangler deploy
        if: github.ref == 'refs/heads/main'
        env:
          CLOUDFLARE_API_TOKEN: ${{ secrets.CLOUDFLARE_API_TOKEN }}
```

## ۱۰. مواردی که اضافه شد (نکات حرفه‌ای از قلم‌افتاده)

این موارد به‌صورت استاندارد برای این نوع پروژه لازم است و به پلن اضافه شدند:

- **بک‌آپ خودکار D1**: خروجی روزانه از دیتابیس (Export) به یک R2 Bucket یا مخزن جدا، برای جلوگیری از از دست رفتن اطلاعات بیماران.
- **Rate limiting و WAF**: استفاده از قابلیت‌های امنیتی رایگان Cloudflare (WAF rules, Bot Fight Mode) برای جلوگیری از حملات brute-force و اسپم فرم‌ها.
- **Captcha روی فرم‌های عمومی** (تماس با ما، رزرو نوبت) با Cloudflare Turnstile (رایگان، جایگزین گوگل ریکپچا).
- **حریم خصوصی داده بیماران**: رعایت اصول حداقلی نگه‌داری داده پزشکی حساس، عدم نمایش اطلاعات بیمار به‌جز برای نقش‌های مجاز.
- **صفحه ۴۰۴ و خطا** اختصاصی و دوزبانه.
- **Health-check Endpoint** برای مانیتورینگ Uptime.
- **اعلان ایمیلی خودکار** برای بیمار هنگام ثبت/تأیید نوبت (نه فقط برای کادر داخلی).
- **پنل گزارش‌گیری ساده** برای دکتر: تعداد نوبت‌های هفتگی/ماهانه، خدمات پرتقاضا.
- **Accessibility پایه** (کنتراست رنگی مناسب، فونت قابل بزرگ‌نمایی) برای بیماران مسن‌تر که مخاطب اصلی فیزیوتراپی هستند.
- **Terms/Privacy Policy** صفحه‌ی ساده حقوقی، حتی در حد یک پاراگراف استاندارد.

## ۱۱. بیزینس پلن تفصیلی

### ۱۱.۱ چشم‌انداز و مأموریت
ایجاد حضور دیجیتال حرفه‌ای برای مطب دکتر کیمیا حبیبی سیاهپوش که اعتماد بیمار را پیش از مراجعه‌ی حضوری جلب کند، فرایند رزرو نوبت را ساده و بدون خطا کند، و زیرساختی مقیاس‌پذیر برای آینده (اپ موبایل) فراهم کند.

### ۱۱.۲ تحلیل بازار و مخاطب
- **بازار هدف اصلی**: ساکنان رباط‌کریم و مناطق همجوار (شهریار، اسلام‌شهر، جنوب غرب تهران) با مشکلات اسکلتی-عضلانی، آرتروز، دیسک کمر، نیاز به فیزیوتراپی و توانبخشی.
- **گروه سنی غالب**: میان‌سال و سالمند (نیاز به UI ساده و خوانا).
- **رقبا**: مطب‌های مشابه که عمدتاً یا سایت ندارند یا فقط از پلتفرم‌های نوبت‌دهی عمومی استفاده می‌کنند بدون هویت برند مستقل؛ مزیت رقابتی ما داشتن سایت اختصاصی + سئوی قوی + پنل داخلی حرفه‌ای است.

### ۱۱.۳ مدل ارزش‌آفرینی
- کاهش تماس تلفنی پشت میز با امکان رزرو آنلاین.
- افزایش نرخ تبدیل بازدیدکننده به بیمار با محتوای اعتمادساز (بیوگرافی، نظرات، تخصص).
- مدیریت داده بیمار در یک سامانه‌ی یکپارچه به‌جای دفترچه کاغذی.

### ۱۱.۴ ساختار عملیاتی
- **Developer**: نگهداری فنی، امنیت، آپدیت.
- **Doctor**: ناظر کلی محتوا و عملکرد، تأیید اخبار، گزارش‌گیری.
- **Secretary**: عملیات روزمره نوبت‌دهی و ثبت بیمار.

### ۱۱.۵ هزینه و زیرساخت (برآورد)
- میزبانی: Cloudflare Workers + D1 + KV — در حد مصرف این پروژه عمدتاً در سقف **پلن رایگان یا Workers Paid ($5/ماه)** قابل پوشش است.
- دامنه: هزینه سالانه ثبت دامنه.
- ایمیل تراکنشی (2FA / اعلان نوبت): پلن رایگان یا ورودی سرویس‌هایی مثل Resend (لایه رایگان تا چند هزار ایمیل در ماه).
- مجموع هزینه ماهانه تخمینی: بسیار پایین (زیر ۱۰ دلار) به‌خاطر معماری Serverless/Edge.

### ۱۱.۶ نقشه راه زمانی (Roadmap)
| هفته | خروجی |

| ۱ | اسکلت Next.js، اتصال D1/KV، صفحات عمومی |

### ۱۱.۷ شاخص‌های موفقیت (KPI)
- تعداد رزرو نوبت آنلاین در ماه
- نرخ تبدیل بازدیدکننده به فرم تکمیل‌شده
